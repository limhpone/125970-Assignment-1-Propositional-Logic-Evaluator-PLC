\section{Key Test Cases}

\textbf{Test 5} (\texttt{t $\vee$ f $\wedge$ f}) validates operator precedence: with $\wedge > \vee$, the expression is parsed as $\texttt{t} \vee (\texttt{f} \wedge \texttt{f}) = \texttt{t}$ (not $(\texttt{t} \vee \texttt{f}) \wedge \texttt{f} = \texttt{f}$). The prefix output \texttt{($\vee$ t ($\wedge$ f f))} confirms correct grouping (see Figure~\ref{fig:ast_example}).

\textbf{Test 6} (\texttt{(t $\vee$ f) $\wedge$ f}) validates that parentheses override default precedence, yielding $(\texttt{t} \vee \texttt{f}) \wedge \texttt{f} = \texttt{f}$ with prefix \texttt{($\wedge$ ($\vee$ t f) f)} (see Figure~\ref{fig:ast_paren}).

\section{Error Handling}

The evaluator handles invalid input at three levels:

\begin{itemize}
    \item \textbf{Lexical errors:} If the input contains an unrecognised character (e.g.\ \texttt{x}), the lexer's \texttt{error()} method raises a \texttt{ValueError} with a descriptive message including the offending character and its position.  This exception propagates through the pipeline and is caught by the GUI's \texttt{evaluate()} method, which displays the error message directly in the truth-value output field.  The user therefore sees, for example, ``\texttt{Error: Illegal character 'x' at position 5}'' instead of a silent failure.
    \item \textbf{Syntax errors:} If the token stream does not conform to the grammar (e.g.\ \texttt{t t} or \texttt{t $\wedge$}), no valid AST is returned (the parser yields \texttt{None} in this implementation), and the GUI displays an error message instead of a result.
    \item \textbf{Runtime exceptions:} Any unexpected exception during the pipeline is caught by the \texttt{evaluate()} method, and the error message is displayed in the truth value output field.
\end{itemize}

This layered approach ensures the application never crashes on malformed input and always provides meaningful feedback to the user.
