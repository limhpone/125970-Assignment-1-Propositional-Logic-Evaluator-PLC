\chapter{IMPLEMENTATION DETAILS}

\section{Lexical Analyser}

The lexical analyser is implemented in \texttt{components/lexica.py} as the class \texttt{PropLogicLexer}, extending \texttt{sly.Lexer}. It converts an input string into a stream of tokens using regular expression pattern matching.

\begin{lstlisting}[caption={Token definitions in \texttt{PropLogicLexer}}, label={lst:lexer}]
class PropLogicLexer(Lexer):
    tokens = { TRUE, FALSE, AND, OR, LPAREN, RPAREN }

    ignore = ' \t'

    AND    = r'[∧&]'
    OR     = r'[∨|]'
    LPAREN = r'\('
    RPAREN = r'\)'
    TRUE   = r't'
    FALSE  = r'f'

    def error(self, t):
        self.index += 1
        print(f"ERROR: Illegal character '{t.value[0]}' "
              f"at line {self.lineno}")
\end{lstlisting}

\textbf{Key design choices:}
\begin{itemize}
    \item Both Unicode ($\wedge$, $\vee$) and ASCII (\texttt{\&}, \texttt{|}) operators are accepted via character class regex patterns.
    \item The \texttt{ignore} attribute silently consumes whitespace.
    \item The \texttt{error()} method reports illegal characters with their line number, advances past the offending character, and continues scanning.
\end{itemize}

\textbf{Tokenisation example.} For the input \texttt{t $\vee$ f $\wedge$ f}, the lexer produces the token stream shown in Table~\ref{table:tokenisation}.

\section{Parser --- LALR(1)}

The parser is implemented in \texttt{components/parsers.py} as the class \texttt{PropLogicParser}, extending \texttt{sly.Parser}. It uses the \textbf{LALR(1)} (Look-Ahead Left-to-right Rightmost-derivation) parsing algorithm~\citep{aho2006compilers}.

\subsection{LALR(1) Parsing Algorithm}

LALR(1) is a \textbf{bottom-up} parsing algorithm that:
\begin{itemize}
    \item Reads input \textbf{left to right} (L)
    \item Produces a \textbf{rightmost derivation} in reverse (R)
    \item Uses \textbf{1 token} of lookahead to make parsing decisions
\end{itemize}

It operates using a \textbf{shift-reduce} mechanism:
\begin{itemize}
    \item \textbf{Shift:} Push the next input token onto the parse stack.
    \item \textbf{Reduce:} When the top of the stack matches the right-hand side of a production rule, replace it with the left-hand side non-terminal and execute the associated \textbf{semantic action} (constructing an AST node).
\end{itemize}

The SLY library automatically generates the LALR(1) parse table from the grammar rules and precedence declarations. A debug file (\texttt{parser.out}) is produced containing the full state machine for inspection.

\subsection{Precedence Declaration}

\begin{lstlisting}[caption={Precedence declaration in \texttt{PropLogicParser}}, label={lst:precedence}]
precedence = (
    ('left', OR),      # Level 1 - lower priority
    ('left', AND),     # Level 2 - higher priority
)
\end{lstlisting}

In SLY, tokens listed \textbf{later} in the precedence tuple bind \textbf{tighter}. This ensures that $\wedge$ has higher precedence than $\vee$. When a shift/reduce conflict arises, the parser compares the precedence of the lookahead token with the precedence of the reduction rule and chooses accordingly.

\subsection{Grammar Rules with Semantic Actions}

Each grammar rule is implemented as a decorated method that constructs the corresponding AST node:

\begin{lstlisting}[caption={Grammar rules with semantic actions}, label={lst:grammar_rules}]
@_('expr OR expr')
def expr(self, p) -> Expression:
    return Expression_logic(Operations.OR, p.expr0, p.expr1)

@_('expr AND expr')
def expr(self, p) -> Expression:
    return Expression_logic(Operations.AND, p.expr0, p.expr1)

@_('LPAREN expr RPAREN')
def expr(self, p) -> Expression:
    return p.expr

@_('TRUE')
def expr(self, p) -> Expression:
    return Expression_bool(True)

@_('FALSE')
def expr(self, p) -> Expression:
    return Expression_bool(False)
\end{lstlisting}

The semantic actions are \textbf{syntax-directed}: each grammar rule directly constructs the corresponding AST node, building the tree bottom-up as reductions occur during parsing.

\caption{LALR(1) shift-reduce parsing trace for \texttt{t $\vee$ f $\wedge$ f}}
\label{table:parse_trace}
\begin{tabular}{@{}clllp{4.5cm}@{}}
\toprule
Step & Stack & Input & Action & Semantic Action \\
\midrule
1  & ---                        & \texttt{t $\vee$ f $\wedge$ f} & Shift \texttt{t}     & --- \\
2  & \texttt{TRUE}              & \texttt{$\vee$ f $\wedge$ f}   & Reduce (5) & \texttt{Expression\_bool(True)} \\
3  & \texttt{expr}              & \texttt{$\vee$ f $\wedge$ f}   & Shift $\vee$       & --- \\
4  & \texttt{expr $\vee$}       & \texttt{f $\wedge$ f}          & Shift \texttt{f}     & --- \\
5  & \texttt{expr $\vee$ FALSE} & \texttt{$\wedge$ f}            & Reduce (6) & \texttt{Expression\_bool(False)} \\
6  & \texttt{expr $\vee$ expr}  & \texttt{$\wedge$ f}            & Shift $\wedge$ $^*$ & --- \\
7  & \texttt{expr $\vee$ expr $\wedge$} & \texttt{f}             & Shift \texttt{f}     & --- \\
8  & \texttt{expr $\vee$ expr $\wedge$ FALSE} & ---               & Reduce (6) & \texttt{Expression\_bool(False)} \\
9  & \texttt{expr $\vee$ expr $\wedge$ expr}  & ---               & Reduce (3) & \texttt{Expression\_logic(AND, \ldots)} \\
10 & \texttt{expr $\vee$ expr}  & ---                             & Reduce (2) & \texttt{Expression\_logic(OR, \ldots)} \\

\bottomrule
\end{tabular}
\end{table}

$^*$ At step 6, a \textbf{shift/reduce conflict} occurs: the parser could either reduce \texttt{expr $\vee$ expr} by rule (2) or shift $\wedge$. Because AND has \textbf{higher precedence} than OR, the parser chooses to \textbf{shift}, correctly binding $\wedge$ tighter.

\section{Evaluation --- Truth Value Computation}

The \texttt{run()} method evaluates the AST by performing a recursive \textbf{bottom-up (post-order) tree traversal}.

\textbf{For leaf nodes} (\texttt{Expression\_bool}):
\begin{equation}
\texttt{run}() = \texttt{value} \in \{\text{True}, \text{False}\}
\end{equation}

\textbf{For binary nodes} (\texttt{Expression\_logic}):
\begin{equation}
\texttt{run}() = \begin{cases}
\texttt{left.run()} \;\wedge\; \texttt{right.run()} & \text{if operation} = \text{AND} \\[6pt]
\texttt{left.run()} \;\vee\; \texttt{right.run()} & \text{if operation} = \text{OR}
\end{cases}
\end{equation}

\textbf{Evaluation trace for \texttt{t $\vee$ f $\wedge$ f}} (using the AST from Figure~\ref{fig:ast_example}):

\begin{enumerate}
    \item Evaluate left child of $\vee$: \texttt{Bool(t)} $\rightarrow$ True
    \item Evaluate right child of $\vee$ (the $\wedge$ subtree):
    \begin{enumerate}
        \item Evaluate left child of $\wedge$: \texttt{Bool(f)} $\rightarrow$ False
        \item Evaluate right child of $\wedge$: \texttt{Bool(f)} $\rightarrow$ False
        \item Compute: $\text{False} \wedge \text{False} = \text{False}$
    \end{enumerate}
    \item Compute: $\text{True} \vee \text{False} = \text{True} \rightarrow$ \texttt{t}
\end{enumerate}

\begin{lstlisting}[caption={Evaluation method in \texttt{Expression\_logic}}, label={lst:run}]
def run(self):
    left_val = self.left.run()
    right_val = self.right.run()
    if self.operation == Operations.AND:
        self.value = left_val and right_val
    elif self.operation == Operations.OR:
        self.value = left_val or right_val
    return self.value
\end{lstlisting}


\section{Translation --- Prefix Notation Generation}

The \texttt{prefix()} method converts the AST into a fully parenthesised prefix notation string via a recursive \textbf{pre-order tree traversal}.

\textbf{For leaf nodes:}
\begin{equation}
\texttt{prefix}() = \begin{cases}
\texttt{"t"} & \text{if value is True} \\
\texttt{"f"} & \text{if value is False}
\end{cases}
\end{equation}

\textbf{For binary nodes:}
\begin{equation}
\texttt{prefix}() = \texttt{"("} \;+\; op \;+\; \texttt{" "} \;+\; \texttt{left.prefix()} \;+\; \texttt{" "} \;+\; \texttt{right.prefix()} \;+\; \texttt{")"}
\end{equation}

where $op \in \{\wedge, \vee\}$ is determined by the node's operation.

\textbf{Translation trace for \texttt{t $\vee$ f $\wedge$ f}}:
\begin{enumerate}
    \item \texttt{prefix}($\vee$ node):
    \begin{enumerate}
        \item Left child: \texttt{prefix}(\texttt{Bool(t)}) = \texttt{"t"}
        \item Right child: \texttt{prefix}($\wedge$ node):
        \begin{enumerate}
            \item Left: \texttt{prefix}(\texttt{Bool(f)}) = \texttt{"f"}
            \item Right: \texttt{prefix}(\texttt{Bool(f)}) = \texttt{"f"}
            \item Result: \texttt{"($\wedge$ f f)"}
        \end{enumerate}
        \item Result: \texttt{"($\vee$ t ($\wedge$ f f))"}
    \end{enumerate}
\end{enumerate}

\begin{lstlisting}[caption={Prefix translation in \texttt{Expression\_logic}}, label={lst:prefix}]
def prefix(self) -> str:
    op_str = '∧' if self.operation == Operations.AND else '∨'
    return f"({op_str} {self.left.prefix()} {self.right.prefix()})"
\end{lstlisting}

\textbf{Key property:} The output is \textbf{fully parenthesised}, making the grouping structure explicit without relying on precedence rules. This ensures the prefix string is an unambiguous representation of the AST.

\subsection{GUI Interaction Flow}

When the user clicks the \textbf{Evaluate} button, the following pipeline executes:

\begin{enumerate}
    \item A fresh \texttt{PropLogicLexer} and \texttt{PropLogicParser} are instantiated.
    \item The input text is tokenised by the lexer.
    \item The parser builds an AST from the token stream.
    \item \texttt{result.run()} computes the truth value.
    \item \texttt{result.prefix()} generates the prefix notation string.
    \item Both results are displayed in the output labels.
    \item If the input is invalid, an error message is shown instead.
\end{enumerate}


\section{Integration of Parser and Translation}

The assignment requires both a \textbf{truth value} and an \textbf{equivalent prefix notation} from a single input expression.  A na\"ive approach would be to evaluate during parsing (embedding computation inside the grammar rules).  Instead, this implementation uses the AST as an \textbf{intermediate representation} that decouples parsing from downstream processing.  The integration works as follows:

\begin{enumerate}
    \item \textbf{Parser $\rightarrow$ AST (syntax-directed translation).}
          Each grammar rule contains a \textbf{semantic action} (Listing~\ref{lst:grammar_rules}) that constructs an AST node.  As the LALR(1) parser performs bottom-up reductions, it simultaneously builds the tree --- this is a form of \emph{syntax-directed translation}~\citep{aho2006compilers}.  The parser's output is the root \texttt{Expression} node.

    \item \textbf{AST $\rightarrow$ Truth value (post-order traversal).}
          The root's \texttt{run()} method (Listing~\ref{lst:run}) recursively evaluates child nodes before the parent, implementing a bottom-up (\emph{post-order}) traversal that mirrors the semantics of propositional logic.

    \item \textbf{AST $\rightarrow$ Prefix string (pre-order traversal).}
          The root's \texttt{prefix()} method (Listing~\ref{lst:prefix}) visits the operator before its children, implementing a top-down (\emph{pre-order}) traversal that directly produces fully parenthesised prefix notation.
\end{enumerate}

\textbf{Why this design?}  Because both \texttt{run()} and \texttt{prefix()} are independent traversals of the \emph{same} AST, the expression is parsed \textbf{only once} and the tree is reused:

\begin{lstlisting}[caption={Pipeline integration in \texttt{MainWindow.evaluate()}}, label={lst:integration}]
result = parser.parse(lexer.tokenize(input_text))  # parse once
truth_value = result.run()                          # traversal 1
prefix_str  = result.prefix()                       # traversal 2
\end{lstlisting}

\textbf{Key insight:} The parser is responsible for \emph{structural analysis} (building the tree with correct precedence and grouping), while translation is a separate phase that \emph{walks} the already-built tree.  This clean separation means adding a new output format (e.g.\ postfix notation) requires only a new traversal method --- the parser and grammar remain unchanged.
